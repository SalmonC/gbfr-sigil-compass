## Key Questions

1. 本轮边界是只交付可实施方案，还是方案确认后继续直接实现 PoC/MVP？
2. Windows UI/运行时是否有硬约束，还是可按本地优先、易部署原则选型？
3. 存档直读涉及未知格式与版本风险时，是否接受“存档解析 PoC 过闸后再决定主入口”的分期策略？
4. 存档解析与求解器是否需要独立进程隔离？
5. 首版是否支持第三方运行时插件，还是仅保留内部可替换接口？
6. 本轮交付到架构文档/ADR，还是同时创建可编译 solution 骨架与依赖守卫测试？

## Decisions Made

| Decision | Rationale |
|---|---|
| 原始诉求单独保存，不与分析稿混写 | 保证需求可追溯，后续修正不会覆盖最初目标 |
| 当前视为绿地项目 | 工作目录为空且不是 Git 仓库，没有既有技术栈可继承 |
| 研究与方案先行，编码须经过关键决策确认 | 存档读取、UI 技术栈和交付边界会显著改变实现成本 |
| 本轮仅交付研究与实现方案 | 用户选择 A，不进入 PoC/MVP 编码 |
| 技术栈无预设约束 | 用户选择 A，允许从 Windows 集成、部署与维护角度选型 |
| 自动采集是硬需求 | 优先验证存档直读；不可行则直接设计 CV，手工录入不作为正式路线 |
| 存档解析使用独立工作进程 | 用户选择 B；隔离不可信二进制输入与第三方解析库，UI/求解器保持简单 |
| 首版不开放第三方插件 | 用户选择 A；保留内部端口，避免首版承担 SDK、安全和兼容性成本 |
| 架构文档与 solution 骨架同步交付 | 用户选择 B；用项目引用和依赖测试执行架构边界 |
| UI 改用 Avalonia（已被替代） | 历史选择；后续 Electron 决策已 supersede，不再作为实现依据 |
| 解析始终使用临时快照，长期备份默认勾选 | 用户选择 A；原存档始终只读，长期备份保存到应用数据目录并受保留策略管理 |
| 未知技能保留但不参与求解 | 用户选择 A；保留 raw ID/hash 和诊断状态，Catalog 更新后可恢复，不允许静默猜测映射 |
| Electron 正式替换 Avalonia | 用户选择 1A；Windows 仍是正式目标，macOS 用于当前调试，C# 领域与只读解析继续复用 |
| 目标编辑器改为“技能池 → 当前激活子目标” | 两侧只做视觉分区、不显示总栏名；一次只激活一个子目标，再从技能池点击添加，从目标项点击删除 |
| 仅“可选目标”显示动态容量 | 必须满足等标题不显示 `n/?`；可选上限为 `24 - 必须满足项数 - 基础能力主词条项数`，所有区域以问号气泡解释用途 |
| 正式 Catalog 与基础设施分闸 | 2.0.2 目录未过发布 P0 时，只允许使用 `test.*` 合成数据验证协议/UI；禁止真实 hash、翻译和 profile 固化 |
| 当前版本 Catalog 采用固定上游快照 | 用户确认短期版本更新不构成价值；开发与发布以已审计 commit 为基线，缺失/低可信数据通过本地 overlay 标注，不阻塞当前离线版本 |
| 发布形态改为便携免安装 | Windows 优先单程序；若 Electron 与 self-contained worker 的资源展开要求使单文件不可靠，则交付封入全部依赖的 zip 解压即用包；macOS 提供正常 `.app` |

## Errors Encountered

| Error | Attempt | Resolution |
|---|---:|---|
| `git status` 返回“not a git repository” | 1 | 记录为绿地项目事实，不初始化仓库，避免未经授权改变版本管理状态 |
| zsh 对不存在的 `LICENSE*` glob 报 `no matches found` | 1 | 改用 `find` 和 GitHub API 核查许可证 |
| 直接打开第三方仓库 `main` 分支路径返回 404 | 1 | 先查询默认分支，再使用实际 `master` 路径 |
| solution 首次编译触发 CA1000/CA1859 | 1 | 保留 warnings-as-errors；将泛型静态工厂移到非泛型 companion，并收紧守卫集合类型 |
| WPF `Application` 与 `GBFRTool.Application` 命名空间冲突 | 1 | 在 App 基类上使用 `System.Windows.Application` 完整限定名 |
| 数据源路由字段触发 CA1859 | 1 | 将只在类内使用的字典字段声明为具体 `Dictionary` 类型 |
| 临时存档审计项目的 FlatSharp 编译器要求 .NET 9 | 1 | 不安装或降级项目运行时；下一次尝试使用 `DOTNET_ROLL_FORWARD=Major` 让隔离工具在现有 .NET 10 上运行 |
| 第三方临时审计器前滚后仍缺少 FlatSharp 编译期/运行期依赖 | 3 | 停止继续修补第三方运行链，转由产品自己的严格只读解析 Worker 对真实快照做契约测试 |
| 恢复会话命令因 `rg --files -g AGENTS.md` 无匹配返回 1，导致后续 `&&` 链未执行 | 1 | 改为分别执行只读检查；确认项目根无 AGENTS.md，仅依赖目录存在无关文件 |
| 第一版求解器按 12,000 状态和每状态 3/10 个候选启发式裁剪，不能证明真正 Top-10 | 2 | 对抗审查构造反例；改为无损 DP、可加随机键和安全状态内 Top-K，并由独立穷举 oracle 验证 |
| 截图测试脚本依赖旧 `/tmp` 解析结果且按显示名断言失败 | 2 | 直接调用产品 Worker 解析用户存档，并按稳定 ID/hash 锁定六域、重复次数和结果指标 |
| 首次发行校验命令包含临时目录递归删除，被安全策略拒绝 | 1 | 使用固定隔离验证目录和可复用 manifest 校验脚本，不在校验命令中执行递归删除 |
| 在仓库根执行 `shasum -c release/SHA256SUMS.txt` 时清单内相对文件名解析到错误目录 | 1 | 进入 `release/` 后复验，两份最终 ZIP 均为 `OK` |
| Phase 11 首次 Forge 打包拒绝仅适用于 noEmit 的 `allowImportingTsExtensions` | 1 | 保留显式测试导入，并启用 TypeScript 6 的相对扩展名重写后重新验证正式 bundle |
| TypeScript 扩展名重写后 Renderer 生成 `.js` 引用但源码仍为 `.ts` | 2 | 为 Renderer Webpack 增加标准 `extensionAlias`，让正式 bundle 与 Node 直接测试共用同一组源码 |
