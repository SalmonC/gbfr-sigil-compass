# 目标配置交互规格

## 1. 页面结构

目标编辑页由左右两个视觉区域组成，但界面不显示“技能池”和“目标技能”这两个总标题。

```text
┌──────────────────────────────┬──────────────────────────────────┐
│ [搜索技能________________]   │ 必须满足                    [?] │
│ 全部 基础能力 攻击 防御 …    │ [伤害上限] [伤害上限] …         │
│                              │                                  │
│ 基础能力                     │ 基础能力主词条              [?] │
│ [攻击力] [体力] [暴击率] …  │ [昏厥] [昏厥] [昏厥] [体力] …   │
│                              │                                  │
│ 攻击                         │ 可选目标                 8/11 [?]│
│ [热血] [背水] [刀上舞] …    │ [狂战士] [斯巴达] [追击] …      │
│                              │                                  │
│ 防御 / 辅助 / 特殊 …         │ 基础词条替代顺序            [?] │
│                              │ 1 [攻击力]  2 [暴击率] …         │
│                              │                                  │
│                              │ 禁止出现                    [?] │
│                              │ [刀上舞] [穷寇心]               │
│                              │                                  │
│                              │ 尽量避开                    [?] │
│                              │ [技能 A] [技能 B] [技能 C]      │
└──────────────────────────────┴──────────────────────────────────┘
```

宽窗口左右并排；窗口变窄时保持技能池可用宽度，右侧目标区允许纵向滚动。首版桌面最小窗口宽度按实际原型测试确定，不依赖横向滚动完成核心操作。

## 2. 添加与删除

右侧一次最多激活一个子目标：

1. 点击未激活子目标的标题、说明或空白区域，激活该子目标并取消其他激活状态。
2. 再次点击当前子目标的非技能区域，退出添加状态。
3. 激活后点击左侧技能，为该子目标添加一次该技能；添加后继续保持激活。
4. 点击右侧已经添加的技能，只删除一个 occurrence，不改变当前激活状态。
5. 未激活任何子目标时点击左侧技能，不修改配置，并在技能池附近提示“请先选择右侧要添加到的区域”。

激活状态使用 2 px 雾金色边框、轻微底色和“正在添加到这里”的文字共同表达，不使用发光、渐变或只靠颜色区分。技能项本身不是激活开关，避免“点击删除”与“切换添加区域”发生冲突。

### 2.1 事件与键盘边界

- 只有子目标容器本身、标题文字和非交互说明可切换激活；帮助按钮、开关、技能项、删除按钮和其他交互后代必须阻止容器激活事件。
- 子目标容器可聚焦；焦点位于容器时，`Enter` 或 `Space` 执行与鼠标相同的激活/取消。
- `Esc` 的处理顺序是：先关闭当前气泡或菜单；没有浮层时才退出添加状态。
- 从左侧添加成功后，键盘焦点保留在原技能项，并通过 `aria-live="polite"` 宣告“已添加到〈区域名〉”。
- 删除后，若同一区域仍有下一项则焦点移动到该项；否则移动到上一项；区域已空时焦点回到区域容器。删除按钮的可访问名称包含技能名和 occurrence 位置。
- 状态机测试必须以同一组用例覆盖鼠标和键盘，不能依赖 React 事件冒泡的默认行为。

## 3. 子目标规则

| 显示名称 | 允许重复 | 顺序有意义 | 可添加范围 | 结果语义 |
|---|---:|---:|---|---|
| 必须满足 | 是 | 是 | 已收录且可参与求解的技能 | 少一次即不显示方案 |
| 基础能力主词条 | 是 | 是 | 基础能力类技能 | 要求出现在因子第一个词条；开启优先保证时，优先级仅次于必须满足 |
| 攻击类主词条 | 是 | 是 | 攻击类技能 | 要求出现在因子第一个词条；开启优先保证时，参与统一主词条完成度比较 |
| 防御类主词条 | 是 | 是 | 防御类技能 | 要求出现在因子第一个词条；开启优先保证时，参与统一主词条完成度比较 |
| 可选目标 | 是 | 是 | 已收录且可参与求解的技能 | 尽量按顺序满足，可牺牲较后目标保证较前目标 |
| 基础词条替代顺序 | 否 | 是 | 基础能力类技能 | 精确主词条不足且允许替代时，按 1、2、3……补位 |
| 禁止出现 | 否 | 否 | 已收录且可参与求解的技能 | 主、副任一词条命中即淘汰整套方案 |
| 尽量避开 | 否 | 否 | 已收录且可参与求解的技能 | 不淘汰方案，只在相同目标完成程度内降低排序 |

“必须满足”和“可选目标”是同一普通目标序列的优先级分段，允许同一技能分别出现在两边；例如截图夹具的伤害上限在前者 3 次、后者 1 次。除此以外，同一技能不能同时占用不同子目标；需要多次时，应在允许重复的同一子目标内添加多次。技能被互斥子目标占用后，左侧仍显示该技能，但在当前激活目标下置灰并说明占用位置。

五个可重复的有序目标默认逐 occurrence 显示，每项都有只在当前草稿内稳定的 `targetEntryId`。`[A, B, A]` 必须保留为三项，删除使用 entry ID 而不是 skill ID。为了密度只允许把相邻相同项压缩成一个 run；run 内仍保留各 entry ID，删除一次固定删除该 run 最后一项。禁止跨越其他技能把同名项全局聚合为 `{skillId, count}`。

## 4. 容量显示

只有“可选目标”标题显示数量和上限，例如 `8 / 11`。其他子目标标题不显示 `8 / ?`、`5 / 5` 等配额徽标。

目标总容量来自最多 12 个因子、每个因子 2 个技能，即 24 个技能位置。高优先级目标先预留位置：

```text
可选目标上限 = 24 - 必须满足数 - 基础主词条数 - 攻击主词条数 - 防御主词条数
```

例如“必须满足”有 8 项，“基础能力主词条”要求 5 项，则“可选目标”最多添加 11 项。这里计算的是配置输入容量，不代表 24 个要求一定能由 12 个现有因子同时满足。

虽然其他标题不展示上限，编辑器仍必须阻止全部目标的总 occurrence 数超过 24；三类主词条目标合计还不能超过 12。达到容量时，左侧技能显示为不可添加，原因是“目标位置已用完（12 个因子 × 每个因子 2 个技能）”。调整任一主词条目标或删除必须满足项后，可选目标上限实时重算。

正常状态显示上限使用 `max(0, 24 - mandatoryCount - basicPrimaryCount - attackPrimaryCount - defensePrimaryCount)`。若加载旧配置后总数超限，保留全部数据并进入显式修复状态：上限显示为 0，另显示“已超出 N 项”，禁止新增、保存和分析；不能出现负数上限，也不能静默删除目标。

## 5. 帮助提示文案

每个子目标标题右侧都有统一的圆形问号图标。鼠标悬停或键盘聚焦时显示气泡；`Esc` 关闭，移开鼠标或焦点离开后关闭。图标使用统一图标库，不使用字符问号或 emoji；视觉尺寸 16–18 px，可交互范围至少 32 px，并提供可访问名称。

| 区域 | 气泡内容 |
|---|---|
| 必须满足 | 这里是方案的底线。任何一个技能没有达到指定次数，方案都不会显示；允许重复添加同一技能。 |
| 基础能力主词条 | 要求这些技能出现在因子的第一个词条。启用“优先保证基础主词条”后，会优先比较已满足数量、精确匹配和替代顺序；未完全满足的方案仍会显示并标出缺少项。允许替代时，会按下方顺序补位。 |
| 攻击类主词条 | 指定攻击类技能出现在因子的第一个词条。开启“优先满足”后，先比较三类主词条的总完成数，再按目标顺序比较精确匹配。 |
| 防御类主词条 | 指定防御类技能出现在因子的第一个词条。开启“优先满足”后，先比较三类主词条的总完成数，再按目标顺序比较精确匹配。 |
| 可选目标 | 角色最多装备 12 个因子，每个因子提供 2 个技能，因此一套目标最多占 24 个技能位置。“必须满足”和三类主词条会占用配置位置；当前还可添加 {remaining} 个可选目标。 |
| 基础词条替代顺序 | 找不到精确的基础能力主词条时，只能从这里选择替代项。顺序越靠前，优先级越高；同一技能只能加入一次。 |
| 禁止出现 | 所选因子的第一个或第二个词条只要包含这里的技能，整套方案就不会显示。 |
| 尽量避开 | 这些技能可以出现，但在目标完成程度相同时，出现次数越少的方案越靠前；不会排在完成了更多目标的方案前面。 |

“可选目标”的 `{remaining}` 必须使用当前实时值。气泡最大宽度 320 px，避免长行跨越主界面；显示延迟约 250 ms，键盘聚焦时不得延迟隐藏。

## 6. 技能池分类与检索

分类 ID、顺序和本地化名称由版本化 Catalog 的 `OrderedFilters` 提供，界面不维护另一份分类 enum 或文案表。当前 Ver. 2.0.2 证据支持的原生顺序为：`Basic Stats → Attack → Defense → Support → Special`，简体中文暂记“基础能力、攻击、防御、辅助、特殊”，发布前仍需由同版本游戏本地化表逐字核验。

“角色”和“+因子”可以作为方便查找的附加筛选，但不能伪装成原生技能类别。未知、误分类或仅有占位哈希的技能进入库存诊断，不进入普通目标选择器和求解器，直到目录通过版本准入。

- 搜索匹配当前语言显示名、稳定英文名和已核准别名。
- 搜索命中的不可选技能仍显示，用户可以查看失效原因。
- 清空搜索后恢复 Catalog 的游戏分类和原始顺序。
- 分类内采用自动换行的紧凑技能卡，不使用下拉框，也不让每个技能独占一整行。
- “禁止出现”和“尽量避开”是无序集合，保存和求解不能受点击先后影响。

## 7. 不可添加状态与冲突修复

可添加性由中央规则服务按当前激活子目标计算。原因按以下优先级只显示最有帮助的一条：

1. 总容量或类别不符，例如“此处只能添加基础能力技能”；
2. 已被其他子目标占用，例如“已在‘禁止出现’中选择”；
3. 当前子目标不允许重复且已经加入，例如“已在‘尽量避开’中选择”。

不可添加项必须降低对比度但仍可读，并同时具有锁定图标、不可添加光标和悬浮/聚焦原因。不能简单使用使控件无法聚焦的原生 `disabled`，否则键盘用户无法得知原因。领域状态必须区分 `CanAdd` 与 `CanRemove`：旧配置中的冲突项始终可从右侧删除，并提供“保留此处、移除其他占用”的原子修复动作；冲突未解决前禁止分析。

关闭“允许基础词条替代”时，替代顺序作为草稿保留但不占用跨区互斥名额。重新开启时重新验证；有冲突则进入上述修复状态，不静默覆盖其他选择。

## 8. 视觉和文本原则

- 使用低饱和夜蓝、象牙白和雾金作为基础色，借鉴原游戏的克制感，但不复制参考站的蓝紫渐变、素材、布局比例或文案。
- “禁止出现”使用深红边框与禁止图标；“尽量避开”使用琥珀色边框与排序向下图标。二者都显示解释文字，不能只靠颜色。
- 圆角、阴影和装饰保持克制；不使用霓虹、玻璃拟态、赛博字体、大面积渐变或泛用 AI 产品式文案。
- 正文使用系统中文字体栈，应用打包后离线可用；所有核心操作提供清晰焦点态。
- UI 只消费 ViewModel 与设计令牌，不直接依赖 Electron IPC、存档模型或求解器 DTO，便于后续整体换肤和单独调试。
