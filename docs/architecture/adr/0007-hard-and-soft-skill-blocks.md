# ADR 0007：分离绝对屏蔽与软屏蔽

- 状态：Accepted
- 日期：2026-07-22

## 背景

部分技能具有不可接受的严重副作用，不能仅靠降低排序处理；其他不理想技能仍可以在库存受限时作为退路。两类规则若共用一个集合，会让求解器无法区分“候选不可行”和“候选较差”。

## 决策

`SkillBlockPolicy` 包含两个无序集合：

- `ForbiddenSkillIds`：所选任一因子的主词条或副词条命中即淘汰候选；
- `SoftBlockedSkillIds`：不影响可行性，按最终出现次数在既有层级降低排序。

绝对屏蔽与必须满足共同组成硬约束层。最终六域为必须满足、可选目标、基础能力主词条、替代顺序和两个屏蔽集合；必须满足与可选目标允许共享 Skill ID，因为它们是同一普通目标序列的优先级分段，其余 14 组跨域组合互斥。请求规范化时保留前四域顺序，只对两个无序集合按稳定 Skill ID 排序。本 ADR 原先的“普通目标 + 必选前缀”存储表述由最终六域合同取代，但其连续优先级语义保留。

UI 使用统一技能池和右侧“禁止出现”“尽量避开”子目标；两者无序、不使用下拉框，具体交互由后续 [目标配置交互规格](../block-pool-ui.md) 修订。分类及顺序由 Catalog 提供并与游戏筛选器一致。某技能被其他选择域占用后不隐藏，而是在原分类位置禁用并展示具体原因。

## 后果

- 比较器版本从 `GBFR-RANK-1` 提升为 `GBFR-RANK-2`。
- 求解器可在建模前删除命中绝对屏蔽的全部因子实例，软屏蔽实例必须保留。
- 旧配置中的单一屏蔽列表迁移为软屏蔽，不能自动升级为绝对屏蔽。
- Profile、分享字符串、request hash 和诊断包分别保存两个集合。
- 请求验证、统一技能池和六个右侧子目标必须共用同一冲突规则服务。
- 未启用的替代池只作为编辑草稿保留，不参与互斥占用和 request hash；重新启用时重新验证。
- 冲突状态区分 `CanAdd` 与 `CanRemove`，旧配置冲突双方始终可删除，避免修复死锁。
- UI 筛选栏使用 Catalog 提供的开放 `FilterCategoryId`；封闭的语义类别只用于 Basic Stats 等业务规则。

## 迁移规则

| 输入状态 | 迁移结果 |
|---|---|
| 旧 `blockedSkillIds=[A,B]` | `SoftBlocked={A,B}`，`Forbidden={}` |
| 旧列表顺序不同 | 按 Skill ID 规范化为同一集合和 hash |
| 旧 blocked 与其他域冲突 | 全部保留为修复草稿，禁止分析，不静默删值 |
| Skill ID 在新 Catalog 缺失 | 标为 orphan，允许移除或映射，不进入普通选择器 |
| 基础属性技能被 Catalog 重分类 | 保留 ID，但基础目标/替代资格失效并要求修复 |
| 已是新 schema | 幂等返回，不重复迁移 |
| 旧分享字符串 | 先按旧 schema 解码，再执行同一迁移 |

旧 `GBFR-RANK-1` 缓存和历史结果保持原版本标签；按新规则重新分析时生成新的 request hash，不能只改显示标签冒充 `GBFR-RANK-2` 结果。

完整语义见 [配装匹配与排序规格](../ranking-specification.md)，界面细节见 [屏蔽池分类多选界面](../block-pool-ui.md)。
