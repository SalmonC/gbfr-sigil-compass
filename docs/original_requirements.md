# GBFR 因子配装方案计算工具——原始诉求

> 记录时间：2026-07-22（Asia/Shanghai）
> 记录原则：以下内容保留用户原始措辞，仅调整为 Markdown 段落，不做需求改写或结论性修正。

我想写一个Windows上使用的碧蓝幻想relink（下称GBFR）的因子配装方案计算工具，其支持以下功能：

0、前置知识：GBFR中一个因子会带有1-2个技能，只带一个技能的因子为垃圾因子，不推荐携带，因此只考虑2技能因子（V+因子）。V+因子的两个词条，在前面的那个词条属于主词条。因子虽然有等级（一般最低为11级，最高为15及），但进行配装时先不优先考虑等级，只要该技能在某个携带的因子中出现一次就算带上一条。技能可以携带多条以提高效果。技能分为主属性、攻击、防御、特殊（记不太清了，从网上搜索资料进行确认）。两个不同的因子可能技能和等级完全相同。

1、读取现持有的因子列表，方式：方案一：直接传入并读取存档文件（不确定是否有加密，去网上看看有无改GBFR存档的项目）；方案二：配合CV图像识别+自动滚动进行读取。优先方案1，不可行则方案2，GitHub的游戏工具应该有很多图像识别的代码，可以参考old-king-wuthering-wave的实现。

2、根据已有的因子，结合手动输入的目标，给出所有能满足方案的配装。配装目标说的3追击+3伤害上限，意思就是有3个因子携带追击技能，3个因子携带伤害上限技能。由于一个因子携带2个技能，达成该配装需要至少3个因子。该工具会输出所拥有的因子能凑出来的配装方案，可能有多种，输出格式如：追击&伤害上限*2 + 伤害上限&追击，意思是3个因子，每个都携带追击和伤害上限技能，其中两个因子的追击放在前面，为主技能，一个因子伤害上限为主技能。如果输出方案超过10个，则只显示前10个。

3、配装时选择目标词条可能少于实际能出的总词条个数，比如12个因子位置能出24个技能，但设置的总目标技能个数为20个，实际就会有4个多余的未指定技能会被携带。可以设置屏蔽技能列表，列出方案时，该方案携带的屏蔽技能越多就越靠后，并且对含有屏蔽技能的条目标红；目标技能有序，排得越靠前的技能越重要；技能选择栏可选：前X条技能为必出，如果未能凑出所有必出技能，则不输出方案。

4、选择目标技能的方式：存在两种可选方式，方式一：通过带搜索功能的下拉栏手动选择技能，一共24个下拉栏，最多选择24个技能，允许重复，最少选择一个，否则开始分析按钮不可点击；方式二：通过一串字符直接输入导入框，将字符解码成技能列表；相对应的，也可将当前已选技能列表导出为字符串，字符串导出的技能顺序有先后之分。可以将当前已选择方案取名并存储在应用内，方便后续持有因子更新后重新分析。

5、输出方案排序：

5.1、满足所有技能目标的情况下，所有因子等级总数越高的越靠前；因子等级也完全一致，则随机排序。

5.2、未能满足所有技能目标，但是能满足必出技能，则满足总技能条数越多越靠前，如果满足条数相同，则非必出技能满足得越靠前的方案在结果列表越靠前（如：目标技能为A、B、C，其中A为必出，方案1出了技能A、B，方案2出了技能A、C，则方案1在结果中更靠前，因为B在目标列表中比C靠前），对比方式为：按目标技能顺序从前往后检索，一直检索到某个技能只在其中一个方案被满足，则该方案更靠前；如果目标技能满足情况完全一致，则同5.1。

5.3、未能满足必出技能，则不输出任何方案，且提示没有满足的方案。

5.4、带有屏蔽技能的方案排在满足目标技能条数相同的正常方案后面，但是排在满足目标技能条数比它少的方案前面。

6、记录落盘原始诉求，然后从第一性原理对需求进行分析，补充可能遗漏的地方，给出实现方案。

*该项目可能会有一些你需要的数据库，如技能列表、因子等，项目链接：https://github.com/choeki/gbfr-relink-sim ；网上还有些项目可以参考，如原神的“莫娜占卜铺”（工具用途接近），GitHub上的 old-king-wuthering-wave，等等，多学习他们的设计思路与实现，但不要直接学习并添加他们的功能，因为游戏不同，工具目标也不同*

## 后续补充需求（2026-07-22 至 2026-07-23）

以下内容是立项后的追加决定，与上面的初始原文分开记录：

- 正式界面使用 Electron，Windows 是发布目标，macOS 用于开发验证。Windows 包封入全部运行库，优先免安装单程序，无法可靠实现时使用解压即用包。
- 存档始终只读。解析应用私有临时快照，不写回游戏目录；应用不提供备份功能，只在读取后建议用户自行备份游戏存档。
- 目标编辑改为统一技能池和多个目标区。先激活目标区，再从技能池添加；点击目标项删除。不可添加的技能保留在原分类中并说明原因。
- 普通目标分为“必须满足”和“可选目标”。基础、攻击、防御三类主词条可分别指定数量和内容；开启优先满足后，优先级仅次于必须满足。只有基础属性主词条支持从有序池中补位。
- 技能屏蔽分为“不能出现”和“尽量避开”。前者命中即淘汰方案，后者只影响同等完成度下的排序。两个集合都按游戏分类展示。
- 命名方案保存在本机，分享字符串包含名称和有序目标，不包含存档或库存。
- 每个目标区提供清空操作。替代主词条区域只在启用补位时显示。
- 分析完成后自动滚动到结果。页面提供目标和结果的快速导航。
- 结果一次显示一套配装，用标签切换前十名。每枚因子单独显示主词条、副词条和等级；替代主词条或带有需避开技能时，标签和结果顶部都要说明。
- 每个方案自动保存目标草稿，并缓存最近一次有效计算。只有成功或无解结果更新缓存；名称变化不使结果过期。目标、库存或占用变化导致结果失效时，直接删除可再生成的旧分析，保留目标方案和已确认配装。
- 结果可以“确认配装”。确认后只占用该结果中的具体因子，其他方案计算时排除这些实例，同词条的其他因子不受影响。
- 两个已确认方案使用同一枚因子时，阻止后确认的操作，必须先取消旧配装。库存更新后找不到原实例时标记缺失，不自动替换。
- 只有开始分析和明确的重新计算会启动求解器。编辑、切换、确认和取消确认都不能暗中计算。
- 目标设计区分成“想要的技能”和“需要避开的技能”两块，用视觉分隔明确层级；原有子目标规则不变。
- 技能池、目标标签和结果因子卡显示技能图标。普通技能优先使用游戏图标；无法取得对应图片时使用明确的本地占位图标，不把角色头像冒充技能图标。
- 提供独立的“已确认配装”入口，可集中查看所有已占用具体因子的配装。名称取确认时的目标方案名，重名依次追加 `-1`、`-2`。
- 已确认配装保存确认时的结果和目标快照。之后修改原目标，不应改变旧配装中的替代提示或问题说明。
- 本地数据必须有明确生命周期：旧版本方案键迁移成功后删除；每个方案最多一份 Top-10 分析；解析用临时存档快照在导入结束后删除；应用不长期保存原始存档副本。
- 求解只在短生命周期 Web Worker 中运行。完成、失败、取消和页面退出都释放 Worker 与 pending request；重复计算的内存增长和真实存档 p95 耗时必须有自动压力测试。
- 24 项目标会出现组合爆炸的合法输入。精确求解设置状态数、候选数和墙钟上限；达到上限时明确停止，不返回近似结果、不写入缓存、也不能确认配装。界面提示用户减少可选目标或增加“不能出现”的技能，禁止以 OOM、白屏或长期无响应的方式失败。
- 提供独立的“持有因子”页面，按具体库存实例显示主副词条、各等级、游戏内装备角色和已确认配装占用；支持名称搜索、技能分类、使用状态筛选和排序。
- 启动时恢复最后一次成功读取后的因子库存快照，但不重新打开原存档。再次手动读取成功后才更新库存快照；上次路径仅用于下一次手动选择文件时的默认位置。
- 窗口变窄时，顶栏操作整体换到下一行，按钮文字不换行、不被压成窄按钮。
- 基础、攻击、防御三类主词条目标合计最多 12 项，并与必须满足、可选目标共同占用 24 项配置容量。

## 后续补充需求（2026-07-24）

- 结果中不属于任何目标的词条要弱化显示，方便快速辨认配装中的附带技能；“尽量避开”的警告样式仍优先。
- 结果中的主、副词条都可手动替换。编辑一个词条时固定另一个词条，只列出存档中真实存在且未被当前结果或其他已确认配装占用的因子。
- 支持替换、删除或添加整枚因子。手动调整结果缓存于原排名方案下，不改变计算排名；确认时占用调整后的具体因子实例。
- 如果两个结果的目标命中情况一致，且差异只来自目标外副词条，则仅在这些目标外词条位于相同主词条位置时合并。位置不同会产生不同的手动替换范围，因此仍作为不同方案展示。
- 结果不显示所有因子的等级总和。因子卡下方改为按技能汇总实际等级，例如两个 15 级追击显示“追击 Lv 30”；手动调整后立即同步。特殊收集型技能显示其实际规则说明。
