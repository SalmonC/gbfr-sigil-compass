# Findings & Decisions

## Phase 15：求解器最优性审阅（完成）

- 当前实现不是启发式“找一个较好结果”，而是按逻辑因子类型枚举数量的有界动态规划；达到状态、变体或 6 秒预算时抛出 `solver.resource_limit`，不会把未完成搜索伪装成最优结果。
- DP 状态由“对目标有用的总词条计数、主词条计数、已用槽位”构成，计数按需求上限截断。最终覆盖向量只依赖这些状态字段，因此状态合并本身具备无损条件。
- 同一状态桶只按可加尾项保留 Top-K：软避开次数、槽位、等级、两个随机稳定系数。后续选择会对桶内每个候选增加相同增量，既有严格次序不会反转；截断点的加法同分全部保留，避免用不具可加性的签名字典序做前缀剪枝。
- 相同有序 `(主词条, 副词条)` 类型的实体先按因子等级降序排列；选择该类型 N 枚时固定取最高等级的 N 枚，所以逻辑方案对应的实体实现达到最高等级和。`A&B` 与 `B&A` 使用不同类型键，不会错误合并。
- token 分配可用交换论证证明：必须满足先消费副词条会最大化剩余主词条数；同一技能的精确主词条目标只争用同一种 token，按位置先取同时最大化完成总数与字典序；所有精确目标处理完再替代，不会让替代抢走精确匹配；可选目标最后按顺序消费，同理得到最大完成数下的最优覆盖向量。
- 不相关双词条因子被删除是安全的：它不能改善任何覆盖，只会增加或保持槽位、软避开和等级之后的尾项；禁止技能因子则按硬约束删除。
- 比较器与 `GBFR-RANK-3` 一致：必选/禁止是可行性，随后为主词条完成数、精确覆盖、基础替代池使用向量、可选完成数、软避开次数、可选位置、槽位、因子等级和、seed 尾项与签名。必须满足列表的排列不进入比较器，只有重复次数有意义。
- `npm run test:solver-oracle` 通过：定向 P0 回归、多重集/方案身份检查及 120 组小规模全量穷举均与求解器返回的完整有序 Top-10 一致。
- 结论：在未触发 `solver.resource_limit` 且输入已通过 schema/Catalog 校验时，返回的 `completed` 结果可视为 `GBFR-RANK-3` 下的全局最优 Top-K；极端合法输入可能直接失败，因此不能宣传为“所有输入必定算出最优解”。
- 现有 oracle 的组合空间枚举独立于 DP，但 token 分配与排序仍是对同一业务规则的第二份实现，不是 ILP/CP-SAT 形式化验证。剩余风险主要是规格解释错误，而非已发现的搜索漏解。
- 三类强制主词条的跨类别顺序被规格固定为“基础→攻击→防御”；如果产品上希望用户跨类别自由排序，需要升级比较器版本，不能视为当前算法错误。

## Requirements

- 目标平台：Windows。
- 对象：GBFR V+ 双技能因子；主/副词条顺序有意义，库存中允许存在属性完全相同的不同实例。
- 输入：优先读取存档；不可行时使用 CV + 自动滚动；后续应考虑保底输入通道。
- 目标：最多 24 个有序技能目标，允许重复；前 X 项可设为必出。
- 求解：最多 12 个因子槽，以技能出现次数满足目标 multiset；可接受部分满足，但必须满足必出目标。
- 排序：必出为硬约束；满足数量优先；同数量时按原 5.4 先比较屏蔽出现次数，再比较目标位置；其后建议依次为少用槽位、因子等级总和、可复现随机打散。
- 输出：最多展示前 10 个方案；同构因子应压缩展示，但库存实例仍需精确区分。
- 配置：技能列表可搜索选择；目标可编码为字符串导入/导出；方案可命名并本地保存。
- 研究：核查技能分类、存档可读性和参考项目；只借鉴与当前目标有关的设计思路。
- 本轮交付边界：研究与实现方案，不直接编码 PoC/MVP。
- 技术栈：无硬约束，由研究结果推荐。
- 自动采集约束：存档直读优先；若不可行，必须转 CV 自动读取；因库存量大，手工录入不现实。

## Research Findings

- 工作区在 2026-07-22 初始为空，且不是 Git 仓库。
- `Nenkai/GBFRDataTools` 含独立 `GBFRDataTools.SaveFile` C# 库，MIT 许可；仓库截至 2026-07-20 仍更新，2026-07-13 发布的工具 release 为 v2.0.0。它证明外层存档可解析，但 release 名不等于已用游戏 Ver. 2.0.2 存档完成因子逐项验证。
- `GBFRDataTools` 主 README 指向已记录的部分 save units，后续必须检查 `SaveFile.cs`、单元 ID 与文档，而不能只凭 README 判定可直接提取因子。
- 用户指定的 `choeki/gbfr-relink-sim` 是带本地化 trait 数据与分享图片导出的 GBFR build simulator；已确认其仓库无许可证、数据仍有临时值和 Ver. 2.0.2 新角色专属缺口，只作结构与核对线索。
- 用户提到的 old-king 项目准确仓库应为 `ok-oldking/ok-wuthering-waves`；它是完整游戏自动化系统，不是专为库存 OCR 的窄工具，借鉴时要限制在窗口定位、截图、OCR/自动滚动等基础设施。
- “莫娜占卜铺”核心仓库为 `wormtql/genshin_artifact`（Rust + Vue + WASM，MIT）；其库存采集通常由独立扫描器 YAS/Amenoma 输出 JSON，再由求解器导入。可借鉴“采集器与求解器解耦”，不能照搬原神伤害/圣遗物模型。
- YAS 相关教程显示大规模库存扫描可通过自动滚动并导出多个标准 JSON 格式完成，1500 件库存约数分钟；这是社区资料而非 GBFR 实测，只能作为 CV 产品流程参考。
- 存档直读具备更具体的结构证据：`SaveData#.dat` 位于 `%LocalAppData%\GBFR\Saved\SaveGames`；`SaveGameFile.FromFile` 以 FlatBuffers 解析两个数据块并校验 XXHash64，而不是常见“加密后不可读”黑盒。
- `SaveIDType.cs` 已命名 `GemManager` 的关键数组：2701 最大槽位、2702 槽位 ID、2703 Gem/Sigil ID hash、2704 技能 1 等级、2706 装备角色、2707 flags；文档显示主体可循环约 5100 项。这已经足以证明库存枚举在结构层面可行。
- 主副技能关联已经由公开代码给出：因子实例 `GemUnitID >= 30000`，主词条 UnitID 为 `120000000 + (GemUnitID-30000)*100`，副词条为主词条 +1，再从 1701/1702 读取哈希与等级。剩余关键未知是 Ver. 2.0.2 真实存档是否全部沿用布局、完整新词条字典和 2707 标志位语义。
- `SaveFile.FromFile` 在哈希失败时会调用内部 `FixCurrentHash` 修改内存模型中的哈希数组（未写回文件）；产品中不应直接复用此副作用语义，应封装严格只读解析器并把 hash mismatch 作为诊断，不得静默修复或写回。
- Nenkai 的 save-unit 文档明确警告仍有许多类型缺失；因此 README/枚举是强可行性证据，但不足以宣称“无需 PoC 即可生产解析”。
- 社区存在存档损坏案例和备份恢复经验；虽然本工具只读不应造成损坏，但实现必须使用共享读/复制快照、绝不覆盖原文件，并提示关闭游戏或进行稳定快照重试。
- 游戏/社区资料把因子类别列为 Basic（基础/基础属性）、Attack、Defense、Support、Special，另有 Character-Specific；用户记忆中的“主属性”应修正为“基础”，不能与“主词条”混称。
- 通用规则是 III+/IV+/V+ 带第二随机 trait、同技能等级可叠加但受最大 trait level 限制；本工具首版按用户要求只计算 V+ 出现次数，不计算实际效果上限，但数据模型应保留等级与 maxLevel，避免未来迁移。
- 2026 年 Ver. 2.0.2 的 Master Traits 会按因子第一词条决定的类别触发收益，因此“主词条顺序”已从展示差异变成真实构筑语义；结果去重绝不能把 `A&B` 与 `B&A` 视为相同。
- `gbfr-relink-sim` 当前 seed（来源标注为 2026-07-16 的 GBFR.PE.Patch.Tool embedded data）含 212 traits、189 sigils、12 个因子槽，并显式建模 `canPrimary`/`canSecondary`、五类 category、secondary pool、主副独立等级。它适合做结构参照和交叉校验，不应直接当作唯一权威数据库。
- `gbfr-relink-sim` 仓库未发现 LICENSE 文件；即使公开，也不能默认复制其源码或整包数据。推荐从用户本机游戏表提取 ID/哈希/名称映射，或仅在许可明确后引入可再分发快照。
- Windows 技术栈初选为 .NET 10 LTS + WPF：截至 2026-07-22，.NET 10 为 active LTS、支持至 2028-11-14；.NET 8 已进入 maintenance 且将在 2026-11-10 结束支持。选 .NET 10 可与现有 C# 存档库同语言、降低 Win32/窗口自动化接入成本并支持 self-contained 发布。WinUI 3/Tauri/Avalonia 不提供足以抵消复杂度的首版收益。
- 求解器候选为 Google OR-Tools CP-SAT 的 .NET NuGet 包：官方支持 C#/.NET、整数约束和优化，适合库存上千且最多 12 槽的有界多重集选择；需用基准测试与一个可核验的小规模穷举 oracle 证明模型正确。
- 持久化候选为 `Microsoft.Data.Sqlite`：本地单文件、官方 .NET provider、适合存库存快照/方案/数据版本；导入导出字符串另使用带 schema/version/checksum 的紧凑格式。
- CV 兜底若触发，可用 Windows.Graphics.Capture 或 Win32 窗口客户区截图；官方捕获 API会显示系统选择 UI/黄色边框，是否影响游戏与自动化体验需 PoC。无论何种截图路径，都必须支持 DPI/分辨率校准与只对前台目标窗口发送滚动。
- **存档路线结论：核心方法明确可行，当前版本仍需过闸。** 1.3.x 容器、实例枚举和主副词条恢复置信度约 95%；Ver. 2.0.2 沿用核心布局的置信度约 80%，必须用 2 份不同玩家的当前存档通过十项 PoC 检查后才可宣称生产兼容。
- 该工具给出了字段关联：因子槽 `GemUnitID >= 30000`；因子 ID/等级分别为 save type 2703/2704；trait hash/level 为 1701/1702；对第 `gemIndex = GemUnitID - 30000` 项，主词条 UnitID 为 `120000000 + gemIndex*100`，副词条为主词条 UnitID + 1。其数据说明称现存与生成存档均验证了 1701/1702。
- 因子库存读取因此不需要从 `Gem ID` 猜副词条；主副词条作为独立 trait 单元保存。读取器只需反序列化 typed tables、按 `(IDType, UnitID)` 建索引，再用上述关系连接。
- `BitterG/GBFR-PE-Patch-Tool` 公开仓库未发现 LICENSE；不可复制其 Go 源码。推荐依赖 MIT 的 Nenkai FlatBuffer/哈希实现，字段编号与关联关系作为互操作事实独立实现，并在代码注释/研究文档中标注来源。
- 1.3 起的 Sigil Synthesis 会把两件 V+ 的四个技能打乱后取两个，因此当前库存可能存在以往掉落池不允许的主副组合；求解器必须信任“存档中实际持有的组合”，不能用掉落规则过滤掉已存在实例。
- 现有因子等级并非只能以一个数概括：存档同时保存 sigil level、primary trait level、secondary trait level。原需求按“因子等级总数”排序时应明确采用哪个字段；建议首版采用 sigil level，并在详情显示三个原始值与异常差异。
- 存档没有发现压缩或加密，只有 XXHash64 完整性校验；读取无需密钥或进程注入。首版只验证校验，不提供重算后写回。
- 官方服务协议对逆向、未经授权的数据操作和影响服务的外部工具有限制；只读本地工具风险低于修改器，但没有官方豁免，不能承诺“零封号风险”。

## Technical Decisions

| Decision | Rationale |
|---|---|
| 将目标技能建模为有序 multiset，而不是 set | 同一技能允许重复，且目标位置决定优先级 |
| 将库存因子建模为实例并另算“展示签名” | 完全相同的因子仍是两个可用物品，但结果展示需聚合为 `*N` |
| 存档导入须保留原文件只读与数据快照 | 降低损坏用户存档的风险，并便于复现解析问题 |
| 存档解析使用 adapter/版本探测层 | 现存库证明容器可读，但保存单元与游戏数据表会随版本演进，不能让 UI/求解器依赖原始 ID |
| 推荐 WPF 而非 Web UI | 本地只读存档、Windows 自动化和 C# 逆向库是核心约束，跨平台不是目标 |
| 推荐 CP-SAT + 小规模穷举 oracle | 直接枚举 `C(n,12)` 不可行；约束模型便于表达重复目标、硬必出、槽位、屏蔽和词典序排序 |
| PoC 通过后存档直读作为唯一首发采集主线 | 当前源码证明核心方法；两份 2.0.2 样本过十项闸门后进入产品，连续两轮核心恢复失败才启动 CV |
| `SaveReader.Worker` 独立进程 | 用户选择 B；损坏输入、FlatBuffers/原生依赖异常不应直接终止 WPF UI |
| 首版仅内部 adapter | 用户选择 A；接口由 Application 定义，Infrastructure 实现，不做反射式插件发现 |
| 本轮创建可编译骨架 | 用户选择 B；架构规则通过项目引用、编译器和自动依赖测试固化 |
| `IInventorySource` 使用通用来源请求 | 防止 Application 被存档路径耦合；Save 使用 file URI，未来 CV 可用窗口定位 URI |
| 导入用例按 SourceId 路由多个数据源 | 新增 CV adapter 时只注册实现，不修改 Application 用例或求解器 |
| 架构守卫不依赖测试框架包 | 骨架阶段保持零额外 NuGet 依赖；守卫直接读取 csproj 并以进程退出码供 CI 使用 |

## User-Confirmed Ranking Decisions (2026-07-22)

- 1B：同等满足、屏蔽和目标位置条件下，使用更少因子的方案优先；用户明确接受这是对原始 5.1/5.2 的补充。
- 2A：该决定现适用于软屏蔽技能：按出现次数计数，`X&X` 贡献 2 次；绝对屏蔽则命中即淘汰，不计排序分。
- 3A：满足目标总条数相同时，先比较屏蔽出现次数，再比较目标位置覆盖向量。
- 4A：等级评分使用所选实体因子的 `sigilLevel` 总和；该分数用于优先选择更少需要强化材料的实现。
- 5：方案等价只看有序 `(primarySkillId, secondarySkillId) -> quantity` 多重集合；等级和物理实例 ID 不参与方案身份。`A&B` 与 `B&A` 是不同方案。相同逻辑方案存在多组物理实例实现时，选择等级评分最高的实例组合，以减少强化材料消耗。

## New Requirement: Basic-Stats Primary Quota (2026-07-22)

- 目标编辑器的 24 个可选位置按 `24-X` 个普通目标位和 `X` 个基础属性主词条目标位展示；24 是容量上限，空目标位仍允许。
- `X` 个专用下拉框只显示版本化 Catalog 判定为 `BasicStats && CanBePrimary` 的技能；目标允许重复。
- 前 N 个必选只作用于普通目标。必选普通技能是硬约束，并在 UI 使用背景、外框、图标和文字醒目标识。
- 开启“优先保证基础属性主词条”后，基础属性目标成为仅次于必选的软目标；未满足仍输出并明确标注。
- 基础属性目标先比较已填充数量，再比较按目标顺序的精确匹配，最后比较有序替代池；可牺牲普通目标来满足该高优先级组。
- 替代只允许来自用户勾选的全局有序池。新勾选项追加到末尾，取消后序号压紧，重新勾选时追加到当前末尾。
- 主词条 token 在普通目标与基础属性目标之间不能双计数；同一因子的另一个副词条仍可满足普通目标。
- 关闭主词条优先后，X 个技能作为普通可选目标处理；替代策略不参与求解。
- Ver. 2.0.2 Basic Stats 明确为攻击力、体力、暴击率、昏厥四项。所有角色在 Insight Rank 3 与 Insight EX 各有一枚相同节点；单枚为每个基础属性主词条因子伤害上限 +20%，最多计 5 个，两枚同时启用时合计每个 +40%、总上限 +200%。
- Master Trait 只按主词条决定的因子类别计数；`Stun&Cap` 计 1，`Cap&Stun` 计 0。
- Attack Power 最大汇总等级在当前抽取数据和 Wiki 间存在 50/45 冲突，Catalog 的 `MaxLevel` 必须允许 null/冲突诊断；该冲突不影响本工具按出现次数求解。

## Architecture Revision: GBFR-RANK-1 (2026-07-22)

- 新增版本化排序规范 `docs/architecture/ranking-specification.md` 和 ADR 0006。
- `BuildRequest` 分离普通目标与 `BasicPrimaryTargetPolicy`，并携带 Catalog 版本。
- `BuildResult` 返回每个基础属性目标的 Exact/Substituted/Missing 解释。
- 分析返回值改为 `BuildAnalysis` 信封，包含 snapshot ID、request hash、比较器版本、run seed 和 solver status，修复独立审阅指出的不可重放问题。
- 新增 `ISkillCatalog` 端口和 `Infrastructure.Catalog.Json` adapter 边界，UI 与求解器不按显示名判断类别。
- seeded tie-break 改为按逻辑因子类型生成可线性化的稳定系数，并由 canonical signature 兜底，避免对完整签名做 CP-SAT 无法优化的黑盒 hash。

## New Requirement: Hard and Soft Skill Blocks (2026-07-22)

- 原单一屏蔽列表拆分为绝对屏蔽与软屏蔽，比较器版本提升为 `GBFR-RANK-2`。
- 绝对屏蔽与必选目标同属可行性硬约束；任一已选因子的主词条或副词条命中一次即淘汰整个候选。

# Final implementation findings (2026-07-22)

- 用户当前提供的 `SaveData1.dat` 可由产品自己的严格只读 Worker 解析为 401 个 V+ 因子；两个备份各 399。主档相对备份新增已知 UnitID 30446/30447。三份校验均通过，损坏副本被拒绝。
- 截图目标中的普通必选库存足够；指定的 3 个昏厥、2 个体力主词条在当前库存最多只能精确满足 1/5，因此 UI 必须保留并解释 4 个缺口。
- 基于完整 signature 的哈希不能用于中间状态 Top-K 裁剪：共同后缀会改变数值顺序。最终采用按逻辑因子类型生成的两组可加系数，并只在完整结果用 canonical signature 兜底。
- 无依据的全局状态上限不能保证真正 Top-10。删除 12,000 状态截断后，以 capped semantic state 做无损 DP；真实样本仅 14,550 状态、约 223 ms。
- 统一 token 分配顺序为 Mandatory → Basic Primary → Optional；Mandatory 优先消费副词条，以保留可用于主词条目标的 token。关闭主词条优先时，原基础目标按顺序接到 Optional 末尾。
- 固定三方 seed 的空类别大部分是角色专属 source quirk；r2 import adapter 将 91 项归入角色专属，并用显式 overlay 处理 6 个普通缺省项及 SUPD 的 Attack 分类纠错。
- Electron 的可靠单文件模式并不能消除 Chromium/.NET 资源目录；Windows 选择 176 MiB 左右的免安装 ZIP，解压后双击 EXE，更易校验和调试。

## Phase 11 findings (2026-07-23)

- 用户给出的参考图强调“一枚因子一张卡”：主词条、副词条和因子等级位于同一视觉单元，空槽只是背景结构。本工具不需要编辑等级和空槽，应保留实例卡片，删除可编辑控件。
- `SolverResult.selected` 已携带 `gemUnitId`、`inventorySlotId` 和各词条/等级。确认配装可以占用具体库存实例；同技能、同等级的其他实例不会被连带占用。
- 缓存是否可复用取决于规范化目标、库存快照、Catalog、排序版本、随机种子和因子占用台账版本。方案改名不应使结果过期；目标、库存或占用变化必须标记旧结果不可直接确认。
- humanizer-zh 已从 `op7418/Humanizer-zh` 安装。界面文本应直接说明动作和后果，避免“基础词条替代顺序”“综合排序第一”等内部模型式表述。
- 推荐标题：`必须满足`、`指定的基础主词条`、`可选目标`、`可接受的替代主词条`、`不能出现的技能`、`尽量避开的技能`。替代说明改为“指定的主词条凑不齐时，可用这里的基础属性补足；越靠前越优先”。
- 未确认的缓存方案与新确认配装共享因子时，不构成资源写冲突：保留缓存、标记受影响结果“已被其他配装占用”，并提供重新计算。两个已确认配装共享实例才是必须阻止的事务冲突。
- `主词条 + 副词条 + 等级 + 同类序号` 适合显示分组，不适合长期占用身份：同类列表前面新增或删除一枚会让后续序号整体漂移。实例占用改用 `gemUnitId + inventorySlotId + 有序主副 hash`，逻辑指纹另行保留。
- 新确认只会从候选集中删除因子。若新占用实例没有出现在某缓存的前十名中，这十名及其顺序不会改变，可直接保留；取消确认会增加候选，可能出现新的更优结果，因此保守要求重算。
- 参考项目固定提交没有许可证。普通技能图标已按来源和 SHA-256 记录用于本地验证；公开发行前仍需明确再分发许可，或改成用户自行导入图标包。
- 角色专属技能在上游大多只映射角色头像，不等于游戏技能 logo。本工具不把头像冒充技能图标，缺图时使用类别占位图。
- 已确认配装必须保存确认时目标快照和结果快照，否则来源方案后续编辑会让旧结果的替代主词条与问题提示发生漂移。

## Phase 12 audit scope (2026-07-23)

- 本轮把“缓存”分为三类分别处理：用户数据（目标方案、已确认配装）、可再生成的分析结果、Electron/Chromium 运行缓存。前一类不是垃圾缓存，不能随失效分析一起删除。
- 失效分析结果应从持久化工作区中删除，而不是只在界面标记过期；库存尚未加载时无法验证库存指纹，因此启动阶段不能据此误删仍可查看的结果。
- 性能审计同时覆盖求解 Worker、Renderer 派生计算、Main/Engine IPC 和持久化；只优化求解器不足以排除界面卡顿或后台对象滞留。

## Phase 12 baseline findings (2026-07-23)

- 当前 macOS 用户数据根目录共约 68 MiB，其中三份长期存档备份占约 66 MiB；两份备份逐字节相同。`Local Storage` 仅约 52 KiB，Code Cache 约 28 KiB，三类 GPU/Dawn 缓存合计约 1.6 MiB。
- 分析结果不是独立文件；每个目标方案最多内嵌一份 Top-10 结果，并与目标、已确认配装一起存入单个 `localStorage` 工作区键。因此不存在“每算一次新增一个缓存文件”，但过期结果当前只被标记，仍会留在工作区 JSON 中。
- 架构文档已经规定长期备份最多 10 份、总量 2 GiB、清单约束和原子写入，但现有 Main 实现只是用时间戳不断复制文件，尚未落实去重、清单和回收。这是实际磁盘增长的首要缺口。
- Renderer 的求解 Worker 在计算结束后继续存活，会长期保留 V8 堆的高水位；页面退出也没有显式终止。`postMessage` 同步失败时，pending promise 也可能留在 Map 中。
- Main 的 Engine 完整性校验用 `readFile` 把每个运行文件整体载入内存后哈希；可改为流式哈希以降低启动峰值。IPC request 写入失败也缺少 pending 清理。
- Renderer 每次目标编辑都会重新计算整份库存指纹，并在方案下拉栏渲染时为每个方案重复过滤库存；数据量当前不大，但属于随方案数线性放大的无效工作。
- 当前运行中的 Electron 进程总 RSS 约 427 MiB（主进程约 134 MiB、Renderer 约 128 MiB、GPU 约 94 MiB、网络进程约 38 MiB、Engine 约 23 MiB）。这只是单点基线，不能据此判定泄漏；需用重复计算后的 GC/进程趋势验证。

## Phase 12 implementation and measurement findings (2026-07-23)

- Cygames 官方站没有给第三方工具分发原始游戏图标的许可，并明确保留复制、再发布和公开传输等权利；Wiki 的 CC 页面许可也明确排除归 Cygames 或其许可方所有的游戏素材。社区项目或 Wiki 只能证明文件来源，不能替代原权利人的授权。
- 缓存生命周期已改为物理移除失效分析载荷：目标计算输入变化立即删除；加载库存后删除指纹不匹配项；确认配装只淘汰使用了新占用实例的缓存；释放占用保守删除可能被新候选超越的缓存。
- Cache 现在记录 `rankingVersion` 与 `requestKey`；旧 v3 缓存缺少版本字段时在加载阶段被丢弃。成功写入 v3 工作区后删除旧 `profiles.v2` localStorage 键。
- 求解 Worker 在完成、失败、取消和页面退出时终止；`postMessage` 同步失败会移除 pending promise。Engine 文件校验改为流式 SHA-256，stdin 写失败会清理 timeout/pending；path grant 增加过期回收和每窗口 8 项上限，同一窗口最多一次存档导入。
- 真实 401 枚 V+ 因子连续求解 15 次：14,550 explored states，p50 179 ms、p95/max 214 ms；每轮强制 GC 后 heap 8.1→8.2 MiB，按一位小数计增长 0.0 MiB，最高 RSS 254.4 MiB。没有观察到随次数递增的求解器堆泄漏。
- 现有备份目录已由新维护器接管：删除一份 SHA-256 完全相同的 23,068,672-byte 重复副本，保留另一份相同内容和一份不同内容；目录从约 66 MiB 降至约 44 MiB，并生成 669-byte 清单。该删除不可从应用内恢复，但相同字节的副本仍保留。
- Electron 使用本地文件页面，不依赖 HTTP 资源；已禁用 HTTP cache 并在启动时清理旧 HTTP cache。Code Cache 和 GPU/Dawn cache 体积很小且有运行性能价值，不做每次强删；它们由 Chromium 自身管理，不与方案缓存混用。
# Phase 13：因子库存、只读快照与分类主词条目标

## 已确认的产品决策

- 应用启动时不重新打开上次的存档，只恢复最后一次成功读取后保存的库存快照。
- 用户再次手动选择并读取存档后，才替换本地库存快照。
- 应用不再创建长期存档备份；界面保留“建议自行备份存档”的简短提示。
- 既有应用备份不保留，备份代码、IPC 参数、测试和用户数据文件应一并清理。
- “持有因子”是独立页面，需支持搜索、分类筛选、排序，并显示装备与已确认配装的占用状态。
- 增加攻击类、防御类主词条目标；基础、攻击、防御主词条与普通目标共同受 24 项总容量约束。

## 架构发现

- 当前 `BuildProfile`、求解器和编辑器把 `basicPrimary` 写成特例。继续复制字段和分配流程会造成三套易分叉的逻辑，应抽象统一的主词条目标分配顺序。
- 基础属性仍有独有的“允许其他基础属性按顺序替代”规则；攻击、防御目标目前只需要精确匹配，不应擅自增加替代池。
- 主进程已掌握存档路径，Renderer 只需获得库存快照。新的快照存储应放在主进程用户数据目录，避免把原存档绝对路径暴露给页面或写入分享字符串。
- 当前 UI 仅在 React 内存保存 `ImportedInventory`，重启即丢失；需要窄 IPC：读取缓存快照、成功导入后原子覆盖快照。
- 当前顶栏虽启用了 `flex-wrap`，但子按钮和标签缺少统一的 `white-space: nowrap` 与不可压缩约束，因此窄窗时仍可能先折断按钮文字。

## 实施与验证结果

- `BuildProfile` 升级到 schema 3，分享前缀和排序版本升级到 `GBFR-RANK-3`；schema 2 的本地方案与分享串会补齐空的攻击、防御主词条字段后继续使用，旧分析缓存失效。
- 三类主词条先统一计算已完成总数，再按基础→攻击→防御的目标顺序比较精确覆盖；基础属性替代池只补基础属性缺口，不会替代攻击或防御目标。
- 应用数据改为单个 `inventory-snapshot.v1.json`。写入采用同目录临时文件加原子 rename；损坏、超大或结构错误的缓存会被忽略。
- Main 只保留路径并把它用于下一次用户主动打开文件选择框；Renderer 收不到绝对路径。启动恢复库存 DTO 时不会访问源存档。
- 应用内备份代码、IPC 参数、维护测试和本机 `save-backups` 目录已删除；启动迁移也会清除旧版本留下的该目录。
- 持有因子页每页只渲染 48 枚实例，401 枚库存不会一次创建全部卡片。可按任一词条名称、库存位置或实例号搜索，按任一词条分类筛选，并查看游戏内装备 ID 与已确认配装名称。
- 720 px 视口下 `bodyScrollWidth === innerWidth`，顶栏按钮计算样式均为 `white-space: nowrap`，没有横向页面溢出；方案操作移至完整的新行。
- 真实 401 枚库存连续求解 15 次的 p50 为 170 ms、p95/max 为 192 ms，post-GC heap 8.1→8.2 MiB，最高 RSS 263.2 MiB。
- 独立复审使用真实库存构造 24 个高频不同可选目标，复现旧 DP 在约 36 秒后增长到 4.3 GiB 并 OOM。当前精确 DP 增加每层 25,000 状态、50,000 候选和 6 秒墙钟上限，Renderer 另设 8 秒终止；超限明确失败，不缓存或确认近似结果。
- 同一 24 项真实回归现在约 143 ms 受控停止，进程 RSS 约 265 MiB；常规截图目标仍精确检索 14,550 状态，15 次 p95 约 178 ms。
- 复审同时发现并修复：128 MiB 以上伪存档在复制前拒绝、旧备份迁移删除降级为 best-effort、Engine Host 不再常驻保存历次库存、解析请求超时与 Worker 20 秒预算对齐、重新导入库存时分页复位。
# Phase 14 UI 重设计发现（2026-07-23）

- 自动设计系统检索推荐了 Data-Dense Dashboard 的密度、筛选、WCAG AA、可见焦点和 150–300ms 反馈；其 Newsletter 结构、蓝橙配色和在线 Inter 字体与本项目不符，明确不采用。
